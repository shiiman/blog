## 1. 背景・課題

公開記事のアイキャッチ差し替えを調査する過程で、Git LFS 追跡対象の画像が「実体（生バイナリ）」のままコミットされている不整合を発見した。

### 根本原因

- `8a483d3 add posts` で記事画像（`posts/**/assets/*.png`）をコミット
- その**後** `035dbac` で `.gitattributes` に `posts/**/assets/*` の LFS パターンを追加
- つまり**画像をコミットした時点ではまだ LFS 追跡対象ではなかった**ため、実体としてコミットされた
- git lfs 自体は正常動作（`filter.lfs.process` 設定済み、LFS フック群も配置済み）

不整合だった 5 枚は `git add --renormalize` で LFS ポインタに正規化し、コミット `2c6f2c9` として push 済み（本設計の対象外＝対応完了）。

### 副次的に発見した問題

- `GEMINI.md` が破損していた（全 2902 行、ユニーク行 44、`## Core Rules` が約 1000 回繰り返し、他セクションが存在しない）。過去の生成事故と推測。既にこの状態でコミット済み。

## 2. 目的

1. LFS 対象ファイルの実体混入を、**ローカルのコミット時点**で確実に防ぐ
2. AI（Claude / Codex / Gemini）が画像を扱う際に事故らないよう、**指示ファイルに運用ルールを明記**する
3. 破損した `GEMINI.md` を正常な構造（`AGENTS.md` 同等）に**再生成**する

## 3. スコープ外（YAGNI）

- 既存履歴に残る実体画像のクリーンアップ（`git lfs migrate` / force push）— `.git` 肥大は許容
- CI（GitHub Actions）による検証 — 今回はローカル pre-commit フックのみ
- `.gitattributes` のパターン拡張（`drafts/`・`svg`/`avif` 等）— 必要になった時点で都度対応

## 5. 詳細設計

### 5.1 `.githooks/pre-commit`

ステージされた Added / Modified ファイルのうち、`.gitattributes` で `filter=lfs` の対象なのに、ステージ内容が LFS ポインタでない（＝実体が混入した）ものを検出し、見つかればコミットを中止する。

```bash
#!/usr/bin/env bash
set -euo pipefail

violations=()
# ステージ済みの Added/Modified を NUL 区切りで列挙（スペース/日本語パス対応）
while IFS= read -r -d '' f; do
  # .gitattributes で filter=lfs の対象か
  [ "$(git check-attr filter -- "$f" | sed 's/.*: //')" = "lfs" ] || continue
  # ステージ内容の先頭行が LFS ポインタ署名でなければ実体混入
  if [ "$(git cat-file blob ":$f" 2>/dev/null | head -1)" != \
       "version https://git-lfs.github.com/spec/v1" ]; then
    violations+=("$f")
  fi
done < <(git diff --cached -z --name-only --diff-filter=AM)

if [ ${#violations[@]} -gt 0 ]; then
  echo "❌ LFS追跡対象なのに実体がステージされています:" >&2
  printf '   - %s\n' "${violations[@]}" >&2
  echo "" >&2
  echo "次で正規化してから再コミットしてください:" >&2
  echo "   git add --renormalize ${violations[*]}" >&2
  exit 1
fi
```

**設計上のポイント / エッジケース**

- シェバンは `#!/usr/bin/env bash`（配列・`read -d ''`・プロセス置換を使うため。macOS の bash 3.2 で動作可）
- 判定は拡張子でなく `git check-attr filter` を使うため、`.gitattributes` の定義と常に一致する
- `--diff-filter=AM` で Added/Modified のみ対象（削除・リネーム旧名は対象外）
- NUL 区切り（`-z` ＋ `read -d ''`）で、`backlog/` に存在するスペース・日本語を含むパスにも対応
- LFS 未インストール環境でも、本フックは core git コマンドのみで動くため機能する（むしろ実体混入を検出できる）

### 5.2 `Makefile`

```makefile
.PHONY: build clean test install-hooks   # install-hooks を追加

install-hooks:
	cp .githooks/pre-commit .git/hooks/pre-commit
	chmod +x .git/hooks/pre-commit
	@echo "pre-commit フックをインストールしました"
```

- git lfs は `pre-commit` フックを使わない（`post-checkout` / `post-commit` / `post-merge` / `pre-push` のみ）ため、`.git/hooks/pre-commit` への配置は既存 LFS フックと干渉しない
- `core.hooksPath` は変更しないので LFS の動作はそのまま

### 5.3 AI 指示ファイル共通セクション（英語）

`CLAUDE.md` / `AGENTS.md` / `GEMINI.md` に以下のセクションを追加する（文面は共通）。

```markdown
## Image Management (Git LFS)

- Images under `backlog/**` and `posts/**/assets/*` are tracked by Git LFS
  (see `.gitattributes`). They MUST be committed as LFS pointers, never as raw binaries.
- After cloning, run `make install-hooks` once to enable the pre-commit guard that
  blocks raw-binary commits of LFS-tracked files.
- When adding images to a NEW location not yet covered by `.gitattributes`, update
  `.gitattributes` FIRST, then commit the images. (Committing before the pattern
  exists is what caused past inconsistencies.)
- If a commit is blocked with "LFS追跡対象なのに実体がステージされています",
  run `git add --renormalize <files>` and commit again.
- Eyecatch images generated by `/eyecatch-create` live at
  `posts/<slug>/assets/eyecatch.png` and are LFS-tracked automatically.
```

## 6. テスト計画（手動）

`make install-hooks` 実行後に確認する。

1. **実体混入を弾く**: 既存のアイキャッチを生バイナリに戻して `git add` → `git commit` → 中止され、違反一覧と `git add --renormalize` 案内が表示される（テスト後は変更を破棄）
2. **正常系**: 正規の LFS ポインタをステージ → コミット成功
3. **対象外**: `.md` など LFS 対象外ファイルのみのコミット → 影響なし（フックは素通り）

## 7. リスク・留意点

- フックは各環境で `make install-hooks` が必要（`core.hooksPath` を使わない設計のため）。README と AI 指示ファイルに明記して周知する
- `GEMINI.md` 再生成時は、内容を `AGENTS.md` / `CLAUDE.md` と突き合わせ、齟齬が無いか確認する
- 本フックはコミットを止めるだけで自動修正はしない（予期せぬ再ステージを避けるため、修正は利用者が明示的に実行する）

## はじめに

このブログ（shiimanblog.com）は、長らく ConoHa WING 上の WordPress で運営してきました。それを **Astro + Cloudflare Pages の静的サイト**へまるごと移行したので、その記録を残しておきます。

移行を考え始めた理由は、ざっくり次のようなものでした。

- **表示速度**を上げたい（静的配信 + CDN）
- **運用コスト**を下げたい（レンタルサーバー → 無料枠で収まるホスティング）
- 記事を **Git で管理**したい（バージョン管理・差分レビュー・ローカル執筆）
- WordPress 本体・プラグイン・PHP の **定期メンテ負荷から解放**されたい

一方で、WordPress には「プラグインを入れれば動く」機能がたくさんあります。問い合わせフォーム、コメント、サイト内検索、広告……。静的サイトに移すということは、これらを **自前で再構築する**ということでもあります。

この記事は、同じように「WordPress から静的サイト / JAMstack へ移りたい」と考えているエンジニア向けに、**何をどう移したか**と **どこでハマったか**を、宣伝抜きで正直にまとめたものです。手順そのものより、「事前に知っておけば消耗しなかった点」に重きを置いています。

## 移行後の構成

まず移行前後の対比から。

| 項目 | 移行前（WordPress） | 移行後（Astro） |
|------|------|------|
| サイト生成 | PHP による動的生成 | Astro（`output: 'static'`）でビルド時に静的化 |
| ホスティング | ConoHa WING | Cloudflare Pages |
| コンテンツ | DB + 管理画面 | Markdown + YAML front matter（Git管理） |
| デプロイ | 管理画面で公開 | `main` に push → GitHub Actions が自動デプロイ |
| 画像 | メディアライブラリ | リポジトリ内に配置（Git LFS 管理） |

Astro 側の要点は2つだけ押さえておけば十分です。

```js
// astro.config.mjs（抜粋）
export default defineConfig({
  output: 'static',        // 全ページをビルド時に静的化
  trailingSlash: 'always', // 旧WordPressのURL（末尾スラッシュ）に揃える
  // ...
});
```

`trailingSlash: 'always'` は地味ですが重要です。WordPress のパーマリンクは `/category/engineering/` のように末尾スラッシュ付きが標準だったので、ここを揃えておかないと後述のリダイレクト設計が崩れます。

記事は次のようなディレクトリ構成で、1記事1ディレクトリに Markdown と画像をまとめています。

```
posts/
└── 2026-06-18_wordpress-to-astro-cloudflare-pages/
    ├── article.md      # 本文 + front matter
    └── assets/
        └── eyecatch.png
```

front matter のスキーマはこんな形です（`src/content.config.ts` を真実の源としてバリデーションしています）。

```yaml
---
title: "記事タイトル"
slug: "url-slug"
date: 2026-06-18T12:00:00.000Z
excerpt: "記事の要約（メタディスクリプションに使用）"
categories: [engineering, wordpress]  # 文字列slugの配列
tags: [wordpress, github]             # 文字列slugの配列
eyecatch: ./assets/eyecatch.png       # 相対パス
draft: false                          # true でビルド対象外
---
```

WordPress 時代は「カテゴリ＝数値ID」で管理されていましたが、Markdown では人間が読める **文字列 slug** に統一しました。slug の一覧は `data/categories.json` / `data/tags.json` で管理し、ビルド時にスキーマでチェックしています。

## WordPressプラグインを「自前」で置き換える

移行のいちばんの山場は、プラグインで賄っていた機能の再実装です。最終的に次の構成に落ち着きました。

| 機能 | WordPress時代 | 移行後 |
|------|------|------|
| 問い合わせフォーム | Contact Form 7 | Cloudflare Pages Functions + Turnstile + Resend |
| コメント | WordPress標準 | Giscus（GitHub Discussions） |
| サイト内検索 | プラグイン | Pagefind |
| アクセス解析 | Site Kit など | Google Analytics 4 |
| 広告 | プラグイン | Google AdSense（自動広告 + インフィード + 記事末尾） |

### 問い合わせフォーム

静的サイトには「サーバーサイド処理」がありません。そこで **Cloudflare Pages Functions**（エッジで動く軽量な関数）を1つ立て、フォーム送信をここで受けます。

- bot 対策に **Cloudflare Turnstile**（reCAPTCHA の代替）を前段に置く
- メール送信は **Resend** の API を叩く

「フォームのためだけにサーバーを維持する」必要がなくなり、関数ファイル1枚で完結するのは静的サイトの気持ちよさのひとつでした。

### コメント・検索

- **コメント**は **Giscus** に置き換えました。コメントが GitHub Discussions に保存されるので、DB もスパム対策プラグインも不要です。
- **サイト内検索**は **Pagefind**。ビルド後の HTML を走査して検索インデックスを生成する仕組みなので、バックエンド不要でサイト内検索が成立します。

いずれも「動的処理をクライアント側 + 外部サービスに寄せる」という、静的サイトの定石どおりの構成です。

## 移行プロセス（フェーズ分け）

行き当たりばったりだと必ず事故るので、フェーズに分けて進めました。

1. **データエクスポート** — WordPress から記事・画像・カテゴリ/タグを書き出し、Markdown へ変換
2. **旧URLの維持** — 301 リダイレクト設計（最重要・後述）
3. **ビジュアルQA** — 本番切替前に新サイトの見た目を作り込み
4. **CI/CD構築** — GitHub Actions でテスト → ビルド → デプロイ
5. **DNSカットオーバー** — ネームサーバーを ConoHa から Cloudflare へ切替
6. **本番動作確認** — リダイレクト・フォーム・コメント・検索・GA・広告を一通り確認

このうち、移行の成否を分けたのは間違いなく **2の「旧URLの維持」**でした。

### 旧URLを1本も落とさない

個人ブログとはいえ、検索流入で読まれている記事があります。URL が変わって 404 を量産すると、積み上げた評価が消えます。なので「**旧URLは1つ残らず新サイトの正しいページへ 301 で飛ばす**」を絶対条件にしました。

Cloudflare Pages では `public/_redirects` というプレーンテキストで宣言的にリダイレクトを書けます。

```
# /旧パス  /新パス  ステータスコード
/feed              /rss.xml          301
/?p=123            /posts/some-slug/ 301
```

ポイントは、リダイレクト表を **手書きしない**ことです。カテゴリ/タグの旧 slug（日本語）と新 slug（英語）の対応から、`_redirects` を **生成するスクリプト**を用意し、`npm run build:redirects` で吐き出すようにしました。さらに、**旧サイトマップの全URLが新サイトでカバーされているか検証するスクリプト**（`npm run verify:redirects`）も自作し、移行漏れを機械的に潰しました。

このリダイレクト生成と検証のロジックは、後述の広告挿入ロジックと並んで **TDD（テスト先行）**で実装しています。「日本語 slug をどうエンコードするか」「子カテゴリのパスはどう組むか」といったルールは口で説明するより**テストで仕様を固定**したほうが圧倒的に安全でした。

## 正直、ここでハマった

ここからが本題かもしれません。移行で実際に踏んだ地雷を、後続の人が同じ穴に落ちないように残しておきます。

### 1. Git LFS の画像が CI で「ポインタ」のままビルドされる

画像は Git LFS で管理しています。ところが GitHub Actions の `checkout` は **デフォルトで LFS の実体を取得しません**。結果、ビルド時に画像が「ポインタファイル（テキスト）」として扱われ、画像処理で落ちました。

```yaml
- uses: actions/checkout@v4
  with:
    lfs: true   # これがないとLFS画像がポインタのままになる
```

ローカルでは普通にビルドできるのに CI だけ落ちる、という厄介なパターンなので注意。

### 2. Astro の `PUBLIC_*` 環境変数は「ビルド時」に焼き込まれる

GA の測定 ID など、クライアントに露出してよい値は `PUBLIC_` プレフィックスの環境変数で渡します。ここで重要なのは、**`PUBLIC_*` はビルド時に静的ファイルへ埋め込まれる**という点です。

`wrangler pages deploy`（デプロイ手順）に渡しても遅く、**ビルドのステップ**に渡さないと値が空のまま出力されます。

```yaml
- name: Build
  run: npm run build
  env:
    PUBLIC_GA_ID: ${{ secrets.PUBLIC_GA_ID }}   # ★ build に渡す（deploy ではない）
```

### 3. `_redirects` の細かい罠（トレイリングスラッシュ・エンコード・親パス）

`_redirects` まわりは、実際のリクエストと1文字単位で合わせる必要があり、ここでいちばん時間を溶かしました。

- **末尾スラッシュの有無は別物**: `/feed` と `/feed/` は別 URL 扱い。両方書く。
- **パーセントエンコードは大文字**: 日本語 slug のエンコードは Cloudflare が受け取る実リクエストに合わせて**大文字**（`%E7%AF%80%E7%B4%84`）で書く。小文字だとマッチしない。
- **子カテゴリは親込みのパス**: 子カテゴリのリダイレクト先は親を含むフルパスにする（例: `/category/fire/savings/`）。

```
# 末尾スラッシュは両方カバーする
/feed              /rss.xml  301
/feed/             /rss.xml  301

# パーセントエンコードは大文字で
/category/%E7%AF%80%E7%B4%84/   /category/savings/   301

# 子カテゴリは親込みパス
/category/savings/   /category/fire/savings/   301
```

このあたりは、まさに前述の「サイトマップ全URLカバー検証スクリプト」が効いた場面です。手で確認していたら確実に取りこぼしていました。

### 4. AdSense はプレビュードメインでは 403 になる

PR を出すと Cloudflare Pages が `https://<branch>.shiimanblog.pages.dev` のプレビュー URL を作ってくれます。便利なのですが、**AdSense はこのプレビュードメイン（`*.pages.dev`）では 403 を返し、広告が表示されません**。AdSense は配信を許可したドメインしか相手にしないためです。

「プレビューで広告が出ない＝バグ」ではないので、広告の最終確認は**本番ドメインで行う**と割り切りました。

### 5. ファビコンが地味に変わっていた

移行作業のどこかでファビコンが差し替わってしまい、ブラウザのタブに見慣れないアイコンが出ていました。旧サイトのファビコンを取り出して復元。地味ですが「読者から見たサイトの同一性」に関わる部分なので、移行チェックリストに入れておくと安心です。

### 6. ドメイン移管の罠（これがいちばん厄介だった）

技術ではなく**手続きの罠**ですが、共有する価値が高いので書いておきます。

ConoHa の「**無料独自ドメイン**」特典で取得したドメインは、次の制約がありました。

- **WINGパックを解約しないとドメインを移管できない**
- 移管に必要な **AuthCode（認証コード）は、サーバー削除後にしか発行されない**

つまり「サーバーを消す → AuthCode が出る → 移管する」という順序で、**サーバーを残したまま安全に移管を済ませる**ことができません。さらにドメインの**更新期限**が近いと、「期限切れリスク」と「サーバー解約のタイミング」を天秤にかけたスケジューリングが必要になります。

> **教訓**: ドメインまわりは、サーバー移行が技術的に終わってからが本番。契約条件（特に無料ドメイン特典の縛り）と更新期限を、移行計画の最初に確認しておくべきでした。

## 移行してよかった点 / 気をつける点

### よかった点

- **表示が速い**: 静的ファイル + CDN 配信。動的生成のオーバーヘッドが消えた。
- **運用コストが下がった**: レンタルサーバー代が不要に。Cloudflare Pages の無料枠で十分回る。
- **記事が Git に乗った**: 差分レビュー、ローカル執筆、`main` への push だけで公開、というシンプルなフロー。PR を出せばプレビュー URL で確認できるのも快適。
- **メンテ負荷からの解放**: PHP やプラグインの脆弱性アップデートに追われなくなった。攻撃対象領域（動的な管理画面・DB）が消えたのも精神的に大きい。

### 気をつける点

- **プラグイン機能は自前**: フォーム・コメント・検索・広告は、外部サービスや自前実装で組み直す前提で計画する。
- **URL維持は最優先 & 機械検証**: 旧URLのリダイレクトは手書きせず、**生成 + 全URLカバー検証**を仕組み化する。SEO 資産を守る生命線。
- **環境変数は「ビルド時」を意識**: 静的サイトでは値が焼き込まれるタイミングが動的サイトと違う。`PUBLIC_*` はビルドステップに渡す。
- **CI 特有の落とし穴**: LFS（`lfs: true`）など、「ローカルでは動くが CI だけ落ちる」系を想定しておく。
- **ドメイン・契約まわりを最初に確認**: 技術より手続きで詰まることがある。特に「無料ドメイン特典」の移管条件と更新期限。

## まとめ

WordPress から Astro + Cloudflare Pages への移行は、「**動的にやっていたことを、ビルド時・外部サービス・エッジ関数へ振り分け直す**」作業の連続でした。手間はかかりますが、終わってみれば表示速度・コスト・運用負荷のすべてで満足しています。

これから移行する人へ、最後に3つだけ。

1. **旧URLのリダイレクトは生成と検証を仕組み化する**（手書きしない）
2. **CI と環境変数の「タイミング」を最初に理解する**（LFS / `PUBLIC_*`）
3. **ドメイン・契約の制約は移行計画の冒頭で確認する**

同じ移行を検討している方の、消耗ポイントを1つでも減らせれば幸いです。
